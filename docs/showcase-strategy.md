# Showcase Strategy

## What Should Be Public

Public:
- Case studies
- Architecture diagrams
- Sanitized screenshots
- Mock demos
- Selected non-sensitive code snippets
- Test results and build notes
- Public demo URLs when available

Private:
- Full source for projects with private workflows
- Credentials, `.env`, browser profiles, real logs
- Real user data, CSV files, DB files, Google Sheet links
- Proprietary workflows or organization-specific details
- Private media, generated face/voice outputs, and copyrighted assets

## Finished Projects

For finished projects, I use one of three publication modes:

1. Public demo + private source
   - Best for personal websites or tools where the user-facing result is enough.

2. Public case study + private source
   - Best for automation, desktop tools, and internal workflow tools.

3. Public sanitized demo source
   - Best when I can replace the real system with mock data and still show the core engineering pattern.

## Unfinished Projects

Unfinished projects should not be hidden completely. They can still show:

- The problem I was solving
- The architecture I explored
- The technical decisions I made
- The parts that already worked
- What I would improve next

This turns a half-finished folder into a credible engineering learning record without pretending it is production-ready.

## Hard-to-Demo Projects

For desktop automation, local AI, extension, and mobile-device tooling, a good portfolio page can include:

- 30-90 second screen recording
- Mock data demo
- Flow diagram
- "What was hard" section
- "What I verified" section
- "What I would do next" section

The reviewer does not need the private system. They need evidence that I can design, build, debug, and explain the system.
